# 契約・ライフサイクルポリシー

資源の獲得と解放、特権的アクセスの境界に関する REJECT/APPROVE 基準を定める。

## 原則

| 原則 | 内容 |
|------|------|
| 対の完結 | 獲得（作成・登録・stage・open）には全経路での解放（削除・解除・cleanup・close）が対応する |
| 全経路 | 成功経路だけでなくエラー経路・早期 return でも対が閉じる |
| 最小境界 | 特権付きアクセスの対象は仕様が許す最小の範囲に絞る |
| 引用義務 | 「確認した」と主張するには獲得側と解放側の両方の位置を引用する |

## REJECT 基準

| 基準 | 判定 |
|------|------|
| 獲得された資源に、いずれかの経路で解放が存在しない | REJECT |
| 認証情報付きリクエストの宛先判定が、仕様に書かれた範囲より広い | REJECT |
| 許可リスト・URL/パス判定が prefix 一致等で意図しない対象を通す | REJECT |
| エラー時に獲得済み資源が残留し、再実行や後続処理に影響する | REJECT |

## APPROVE 基準

| 基準 | 判定 |
|------|------|
| 全ての獲得に全経路の解放が対応し、行を引用できる | APPROVE |
| 特権付きアクセスの境界が仕様の記述と一致する | APPROVE |
