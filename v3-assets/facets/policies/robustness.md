# ロバストネスポリシー

失敗経路の設計・入力検証の強度・権限衛生に関する REJECT/APPROVE 基準を定める。

## 原則

| 原則 | 内容 |
|------|------|
| 部分失敗の隔離 | 複数項目処理での単一項目の失敗は、仕様に明記がない限り全体を失敗させない |
| 検証の完全形 | 形式検証は仕様が定める完全な形で行う。省略形は検証ではない |
| 早期検証 | サイズ・型・内容の検証は資源を消費する前に行う |
| 秘匿衛生 | 秘匿情報に触れる資源は権限を明示し、ログ・エラーメッセージへ漏らさない |

## 失敗経路

| 基準 | 判定 |
|------|------|
| 単一項目の失敗が仕様の明記なく処理全体を失敗させる | REJECT |
| 失敗項目を警告つきでスキップし、残りが継続する | APPROVE |
| エラーを握りつぶして成功と区別できない状態にする | REJECT |
| 失敗時のメッセージから原因の特定に必要な情報が欠落する | REJECT |

## 入力検証

| 基準 | 判定 |
|------|------|
| 形式署名の照合が仕様の完全形に満たない（例: PNG signature 8バイト中4バイトのみ） | REJECT |
| サイズ上限の検証が全量読み込みの後にある | REJECT |
| 外部入力の種別判定が拡張子や自己申告ヘッダだけに依存する | REJECT |

## 権限と秘匿

| 基準 | 判定 |
|------|------|
| 秘匿情報を含み得るファイルを権限指定なしで作成する | REJECT |
| 認証情報が既存の慣例経路と異なる方法で取得・使用される（妥当性の説明なし） | REJECT |
| 認証情報・トークンがログやエラーメッセージに出得る | REJECT |
