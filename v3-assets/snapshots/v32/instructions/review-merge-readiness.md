# Merge Readiness Review

専門レビュー後の累積差分が、今後保守されるコードベースに品質面で入れてよい状態かレビューしてください。

手順:
1. Knowledge と Policy の Source Path が渡されている場合は開き、`##` セクションの基準を確認する
2. タスクの元要件、計画、変更差分、実行証跡、前段レビューを確認する
3. 要求未達、既存動作の破壊、テスト不足、要求外変更、保守困難化、明らかなリスクを探す
4. 差分で追加・変更された ID、名前、メタデータ、設定、環境変数、出力契約、型フィールド、status/discriminant、helper、adapter、entrypoint を列挙する
5. 列挙した項目ごとに、同種箇所、mock、fixture、factory、test double、保存・表示・実行入口を Grep / Glob / Read で確認し、確認した検索語とファイルを証跡に残す
6. 副作用や状態変更は、正常終了・早期終了・例外・中断・cleanup / rollback の経路を分けて確認する
7. 将来の保守者が変更理由、影響範囲、検証方法を追跡できるか確認する
8. `resolved` 判定は修正差分ではなく、元 finding の期待結果・受入条件・元要件へ戻って確認する
9. マージを止めるべき品質・保守性の問題だけを、場所・影響・修正方針付きで指摘する

10. **要求トレース表**を作成する。タスクの元要件（発注書）の「期待する挙動」を1項目ずつ列挙し、各項目に (a) 実装の位置（ファイル:行） (b) その挙動を検証するテストの位置、の両方を引用する。どちらか一方でも欠ける項目が1つでもあれば REJECT とする。表はレポートに必ず含める
