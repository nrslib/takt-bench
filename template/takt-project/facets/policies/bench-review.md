# ベンチレビューポリシー

テストスイートと仕様を唯一の正として、実装の合否を客観的に判定する。

## 原則

| 原則 | 基準 |
|------|------|
| テストは仕様 | テストコードの変更・スキップは無条件で REJECT |
| 全テスト成功が前提 | 1 件でも失敗があれば REJECT |
| 客観性 | 指摘には根拠（場所・仕様の該当箇所）を必須とする |
| 過剰指摘の禁止 | スタイルの趣味・仕様外の要求で REJECT しない |
| 収束優先 | 解消済みの指摘を蒸し返さない |

## 判定基準

| 基準 | 判定 |
|------|------|
| テストが 1 件以上失敗 | REJECT |
| テストコードが変更・削除・スキップされている | REJECT |
| テストを通すためのハードコード・入力への特別分岐がある | REJECT |
| 型チェックが失敗する | REJECT |
| 仕様（README）と明確に矛盾する挙動がある | REJECT |
| 到達不能コード・未使用コードが追加されている | REJECT |
| 上記に該当せず、軽微な改善余地のみ | APPROVE |
| 問題なし | APPROVE |

## 検証手順の要件

- テストと型チェックは推測せず実際に実行して結果を確認する
- 差分だけでなく、差分が依存する周辺コードも確認する
